# Kmesh Dashboard 技术实现说明

本文档介绍 Kmesh Dashboard 后端的技术实现，重点说明后端与 Kubernetes 集群的交互方式，以及 Waypoint、熔断、限流等模块的 API 如何应用到集群中。

---

## 1. 概述

Dashboard 后端采用 Go 实现，通过以下客户端与集群交互：

| 客户端 | 用途 |
|--------|------|
| **kubernetes.Interface** | 访问 Node、Service、Pod 等标准 K8s 资源 |
| **gatewayapiclient.Interface** | 访问 Gateway API（Waypoint 即为特定 Gateway） |
| **dynamic.Interface** | 访问 Istio CRD（DestinationRule、EnvoyFilter 等） |

集群配置通过 `KUBECONFIG` 或集群内 InCluster 方式加载：

```go
// internal/k8s/config.go
func GetConfig(kubeconfigPath string) (*rest.Config, error) {
    if kubeconfigPath != "" {
        return clientcmd.BuildConfigFromFlags("", kubeconfigPath)
    }
    if path := os.Getenv("KUBECONFIG"); path != "" {
        return clientcmd.BuildConfigFromFlags("", path)
    }
    return rest.InClusterConfig()
}
```

所有 API 请求经 JWT + Casbin RBAC 鉴权，写/删操作会记审计日志。

## 3. Waypoint 安装与管理

Waypoint 本质上是 **Gateway API** 中的 `Gateway` 资源，通过 `gatewayClassName: istio-waypoint` 标识。

### 3.1 资源与字段映射

| API 请求字段 | 是否生效 | Gateway 资源/集群操作 | 说明 |
|-------------|----------|------------------------|------|
| `namespace` | ✓ | `metadata.namespace` | 命名空间 |
| `name` | ✓ | `metadata.name` | Waypoint 名称，默认 `waypoint` |
| `trafficFor` | ✓ | `metadata.labels[istio.io/waypoint-for]` | 流量类型：`service` / `workload` / `all` / `none`，空则不设置 |
| `revision` | ✓ | `metadata.labels[istio.io/rev]` | Istio Revision |
| `proxyImage` | ✓ | `metadata.annotations[sidecar.istio.io/proxyImage]` | Waypoint 代理镜像 |
| `enrollNamespace` | ✓ | Namespace `metadata.labels[istio.io/use-waypoint]` | 为命名空间打标签，使该命名空间下 Pod/Service 使用该 Waypoint |
| `overwrite` | ✓ | — | 若已有同名 Waypoint（`gatewayClassName: istio-waypoint`），先删除再创建 |
| `waitReady` | ✓ | — | 创建后轮询 Gateway Status（每 2s），直到 `Programmed=True` 或超时 60s 后返回 |
| — | — | `spec.gatewayClassName` | 固定 `istio-waypoint` |
| — | — | `spec.listeners` | 固定 mesh 监听器：port 15008，协议 HBONE |

后端通过 Gateway API Client 调用 `Create()`，等价于向集群 apply 如下 YAML。例如请求 `{ "namespace": "default", "name": "waypoint", "trafficFor": "service", "proxyImage": "ghcr.io/kmesh-net/waypoint:latest" }` 时：

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: waypoint
  namespace: default
  annotations:
    sidecar.istio.io/proxyImage: ghcr.io/kmesh-net/waypoint:latest
  labels:
    istio.io/waypoint-for: service
spec:
  gatewayClassName: istio-waypoint
  listeners:
    - name: mesh
      port: 15008
      protocol: HBONE
```

若不填 `trafficFor`、`revision`，则对应 label 不会出现；`proxyImage` 为空时使用默认 `ghcr.io/kmesh-net/waypoint:latest`。

### 3.2 列表接口 `GET /api/waypoint/list`

通过 Gateway API Client 列出指定命名空间（或全部）的 Gateway，过滤 `gatewayClassName == "istio-waypoint"`：

```go
gwList, err := gwClient.GatewayV1().Gateways(ns).List(r.Context(), listOpts)
// 过滤
for _, gw := range gwList.Items {
    if gw.Spec.GatewayClassName != "istio-waypoint" {
        continue
    }
    // 提取 trafficFor、revision、Programmed 状态等
}
```

响应字段来自：

- `gw.Namespace`, `gw.Name`
- `gw.Labels["istio.io/waypoint-for"]` → `trafficFor`
- `gw.Labels["istio.io/rev"]` → `revision`
- `gw.Status.Conditions` 中 `GatewayConditionProgrammed` → `programmed`

### 3.3 安装接口 `POST /api/waypoint/apply`

根据请求构建 Gateway 并 Create 到集群：

```go
gw := &gatewayv1.Gateway{
    ObjectMeta: metav1.ObjectMeta{
        Name:      req.Name,
        Namespace: req.Namespace,
        Annotations: map[string]string{
            "sidecar.istio.io/proxyImage": img,  // 默认 ghcr.io/kmesh-net/waypoint:latest
        },
    },
    Spec: gatewayv1.GatewaySpec{
        GatewayClassName: "istio-waypoint",
        Listeners: []gatewayv1.Listener{{
            Name:     "mesh",
            Port:     15008,
            Protocol: gatewayv1.ProtocolType("HBONE"),
        }},
    },
}
if req.TrafficFor != "" {
    gw.Labels["istio.io/waypoint-for"] = req.TrafficFor
}
if req.Revision != "" {
    gw.Labels["istio.io/rev"] = req.Revision
}
gwClient.GatewayV1().Gateways(req.Namespace).Create(r.Context(), gw, metav1.CreateOptions{FieldManager: "kmesh-dashboard"})
```

应用后，Istio/Kmesh 会根据该 Gateway 创建 Waypoint 工作负载，状态体现在 `Gateway.Status.Conditions` 中。

### 3.4 删除接口 `POST /api/waypoint/delete`

按 `namespace` + `names` 删除 Gateway；`names` 为空时列出该命名空间下所有 Waypoint Gateway 并全部删除。

## 4. 熔断配置

熔断通过 **Istio DestinationRule** 的 `trafficPolicy.connectionPool` 实现，使用 `dynamic.Interface` 操作 CRD。**熔断策略作用于 Waypoint**，下发前需确保目标命名空间已安装 Waypoint。

### 4.1 预设模板（前端）

前端提供三种预设模板，选择后自动填充表单数值；后端不接收 `preset` 字段，只接收填充后的具体阈值。

| 模板 | maxConnections | maxPendingRequests | maxRequests | maxRetries | connectTimeoutMs | maxRequestsPerConnection |
|------|----------------|--------------------|-------------|------------|------------------|---------------------------|
| **保守** (conservative) | 10 | 20 | 50 | 2 | 500 | 1 |
| **标准** (standard) | 100 | 200 | 500 | 3 | 1000 | 2 |
| **激进** (aggressive) | 1000 | 2000 | 5000 | 5 | 3000 | 10 |

定义位置：`frontend/src/types/circuitbreaker.ts` 中的 `CIRCUIT_BREAKER_PRESETS`。

### 4.2 资源与字段映射

**GVR**（Group-Version-Resource）：`networking.istio.io/v1beta1` / `destinationrules`

| API 请求字段 | DestinationRule 路径 | 说明 |
|-------------|----------------------|------|
| `host` | `spec.host` | 目标服务，如 `reviews.default.svc.cluster.local` |
| `maxConnections` | `spec.trafficPolicy.connectionPool.tcp.maxConnections` | TCP 最大连接数 |
| `connectTimeoutMs` | `spec.trafficPolicy.connectionPool.tcp.connectTimeout` | 连接超时，如 `100ms`、`2s` |
| `maxPendingRequests` | `spec.trafficPolicy.connectionPool.http.http1MaxPendingRequests` | HTTP/1.1 最大挂起请求数 |
| `maxRequests` | `spec.trafficPolicy.connectionPool.http.http2MaxRequests` | HTTP/2 最大并发请求数 |
| `maxRetries` | `spec.trafficPolicy.connectionPool.http.maxRetries` | 最大重试次数 |
| `maxRequestsPerConnection` | `spec.trafficPolicy.connectionPool.http.maxRequestsPerConnection` | 每连接最大请求数 |

后端通过 Dynamic Client 调用 `Create()` 或 `Update()`，等价于向集群 apply 如下 YAML。例如请求 `{ "namespace": "default", "name": "reviews-cb", "host": "reviews.default.svc.cluster.local", "maxConnections": 100, "maxPendingRequests": 50, "maxRequests": 2, "maxRetries": 3, "connectTimeoutMs": 500 }` 时：

```yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: reviews-cb
  namespace: default
spec:
  host: reviews.default.svc.cluster.local
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
        connectTimeout: 500ms
      http:
        http1MaxPendingRequests: 50
        http2MaxRequests: 2
        maxRetries: 3
```

若某字段未填或为 0，则对应配置项不会出现在 YAML 中。

### 4.3 下发前置条件（Waypoint 检查）

在创建/更新 DestinationRule 前，后端会调用 `HasWaypointInNamespace()` 检查目标命名空间是否存在 Waypoint（`gatewayClassName: istio-waypoint`）：

- 若不存在 Waypoint，返回 `412 Precondition Failed`，提示「熔断策略作用于 Waypoint，请先在命名空间 xxx 安装 Waypoint」；
- 若存在，则继续构建并下发 DestinationRule。

### 4.4 构建 DestinationRule

请求转换为 Unstructured 后创建/更新：

```go
// buildDestinationRule 核心逻辑
spec := map[string]interface{}{"host": req.Host}
tcp := map[string]interface{}{}
http := map[string]interface{}{}
if req.MaxConnections > 0 {
    tcp["maxConnections"] = req.MaxConnections
}
if req.ConnectTimeoutMs > 0 {
    tcp["connectTimeout"] = formatMs(req.ConnectTimeoutMs)  // "100ms" 或 "2s"
}
if req.MaxPendingRequests > 0 {
    http["http1MaxPendingRequests"] = req.MaxPendingRequests
}
if req.MaxRequests > 0 {
    http["http2MaxRequests"] = req.MaxRequests
}
// ... 其他字段
spec["trafficPolicy"] = map[string]interface{}{"connectionPool": map[string]interface{}{"tcp": tcp, "http": http}}

// 创建到集群
dyn.Resource(destinationRuleGVR).Namespace(req.Namespace).Create(...)
```

若资源已存在，则 `Update` 覆盖。Istio 下发该 DestinationRule 到 Envoy，对 `spec.host` 对应上游应用连接池与熔断策略。

### 4.5 列表接口

遍历命名空间内所有 DestinationRule，提取含 `trafficPolicy.connectionPool` 的条目并解析上述字段返回。

## 5. 限流配置

限流通过 **Istio EnvoyFilter** 注入 `envoy.filters.network.local_ratelimit` 实现，采用 Token Bucket 算法。**限流策略作用于 Waypoint**，下发前需确保目标命名空间已安装 Waypoint。

### 5.1 资源与字段映射

**GVR**: `networking.istio.io/v1alpha3` / `envoyfilters`

| API 请求字段 | EnvoyFilter 路径 | 说明 |
|-------------|-------------------|------|
| `statPrefix` | `configPatches[].patch.value.typed_config.stat_prefix` | 统计前缀 |
| `maxTokens` | `typed_config.token_bucket.max_tokens` | 桶容量 |
| `tokensPerFill` | `typed_config.token_bucket.tokens_per_fill` | 每次填充令牌数 |
| `fillIntervalSec` | `typed_config.token_bucket.fill_interval` | 填充间隔，如 `60s` |
| `workloadSelector` | `spec.workloadSelector.labels` | 作用工作负载标签 |

后端通过 Dynamic Client 调用 `Create()` 或 `Update()`，等价于向集群 apply 如下 YAML。例如请求 `{ "namespace": "default", "name": "reviews-ratelimit", "statPrefix": "local_rate_limit", "maxTokens": 100, "tokensPerFill": 10, "fillIntervalSec": 60, "workloadSelector": { "app": "reviews" } }` 时：

```yaml
apiVersion: networking.istio.io/v1alpha3
kind: EnvoyFilter
metadata:
  name: reviews-ratelimit
  namespace: default
spec:
  workloadSelector:
    labels:
      app: reviews
  configPatches:
    - applyTo: NETWORK_FILTER
      match:
        listener:
          filterChain:
            filter:
              name: envoy.filters.network.tcp_proxy
      patch:
        operation: INSERT_BEFORE
        value:
          name: envoy.filters.network.local_ratelimit
          typed_config:
            "@type": "type.googleapis.com/envoy.extensions.filters.network.local_ratelimit.v3.LocalRateLimit"
            stat_prefix: local_rate_limit
            token_bucket:
              max_tokens: 100
              tokens_per_fill: 10
              fill_interval: 60s
```

若 `workloadSelector` 为空，则 `spec.workloadSelector` 不出现，策略作用于命名空间内所有工作负载。`maxTokens`、`tokensPerFill`、`fillIntervalSec` 未填时使用默认值 4、4、60s。

### 5.2 ConfigPatch 结构

限流 filter 通过 `configPatches` 在 `envoy.filters.network.tcp_proxy` 前插入：

```go
value := map[string]interface{}{
    "name": "envoy.filters.network.local_ratelimit",
    "typed_config": map[string]interface{}{
        "@type":       "type.googleapis.com/envoy.extensions.filters.network.local_ratelimit.v3.LocalRateLimit",
        "stat_prefix": statPrefix,
        "token_bucket": map[string]interface{}{
            "max_tokens":      req.MaxTokens,      // 默认 4
            "tokens_per_fill": req.TokensPerFill,  // 默认 4
            "fill_interval":   fillInterval,       // 如 "60s"
        },
    },
}
spec["configPatches"] = []interface{}{
    map[string]interface{}{
        "applyTo": "NETWORK_FILTER",
        "match": map[string]interface{}{
            "listener": map[string]interface{}{
                "filterChain": map[string]interface{}{
                    "filter": map[string]interface{}{"name": "envoy.filters.network.tcp_proxy"},
                },
            },
        },
        "patch": map[string]interface{}{
            "operation": "INSERT_BEFORE",
            "value":    value,
        },
    },
}
```

`workloadSelector` 用于限定仅对匹配 label 的 Pod 生效。

### 5.3 下发前置条件（Waypoint 检查）

在创建/更新 EnvoyFilter 前，后端会调用 `HasWaypointInNamespace()` 检查目标命名空间是否存在 Waypoint：

- 若不存在 Waypoint，返回 `412 Precondition Failed`，提示「限流策略作用于 Waypoint，请先在命名空间 xxx 安装 Waypoint」；
- 若存在，则继续构建并下发 EnvoyFilter。

### 5.4 应用到集群

```go
dyn.Resource(envoyFilterGVR).Namespace(req.Namespace).Create(r.Context(), ef, ...)
```

已存在时执行 `Update`。Istio 将 EnvoyFilter 下发到匹配的 Envoy，在 TCP 代理前注入 local_ratelimit，超出令牌限制的连接会被拒绝。

### 5.5 列表接口

遍历 EnvoyFilter，仅保留 `configPatches` 中含 `envoy.filters.network.local_ratelimit` 的条目，解析 `token_bucket` 与 `workloadSelector` 返回。

## 6. 认证策略

认证策略（授权策略）通过 **Istio AuthorizationPolicy** 实现网格内访问控制，用于限制哪些来源可以访问指定工作负载的哪些操作。使用 `dynamic.Interface` 操作 CRD。

### 6.1 资源与字段映射

**GVR**（Group-Version-Resource）：`security.istio.io/v1beta1` / `authorizationpolicies`

| API 请求字段 | AuthorizationPolicy 路径 | 说明 |
|-------------|--------------------------|------|
| `namespace` | `metadata.namespace` | 命名空间 |
| `name` | `metadata.name` | 策略名称 |
| `action` | `spec.action` | 动作：`ALLOW` / `DENY`，表示匹配规则时允许或拒绝 |
| `selector` | `spec.selector.matchLabels` | 工作负载选择器，限定策略作用的目标 Pod（按 label 匹配） |
| `rules` | `spec.rules` | 规则列表，每条规则包含 `from`（来源条件）与 `to`（操作条件） |

**规则 `from`（来源）**：

| 子字段 | AuthorizationPolicy 路径 | 说明 |
|--------|---------------------------|------|
| `ipBlocks` | `rules[].from[].source.ipBlocks` | IP 段，如 `10.0.0.0/8` |
| `namespaces` | `rules[].from[].source.namespaces` | 来源命名空间列表 |
| `principals` | `rules[].from[].source.principals` | 身份主体，如 `cluster.local/ns/default/sa/sleep` |

**规则 `to`（操作）**：

| 子字段 | AuthorizationPolicy 路径 | 说明 |
|--------|---------------------------|------|
| `hosts` | `rules[].to[].operation.hosts` | 目标主机列表 |
| `ports` | `rules[].to[].operation.ports` | 目标端口列表 |
| `paths` | `rules[].to[].operation.paths` | HTTP 路径，如 `/api/*` |
| `methods` | `rules[].to[].operation.methods` | HTTP 方法，如 `GET`、`POST` |

后端通过 Dynamic Client 创建/更新 AuthorizationPolicy，等价于向集群 apply 如下 YAML。例如请求 `{ "namespace": "default", "name": "reviews-allow", "action": "ALLOW", "selector": { "app": "reviews" }, "rules": [{ "from": [{ "source": { "namespaces": ["default", "sleep"] } }], "to": [{ "operation": { "paths": ["/api/*"], "methods": ["GET"] } }] }] }` 时：

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: reviews-allow
  namespace: default
spec:
  action: ALLOW
  selector:
    matchLabels:
      app: reviews
  rules:
    - from:
      - source:
          namespaces:
            - default
            - sleep
      to:
      - operation:
          paths:
            - /api/*
          methods:
            - GET
```

若某字段未填或为空，则对应配置项不会出现在 YAML 中。`action` 默认 `ALLOW`；`selector` 为空时作用于命名空间内所有工作负载；`rules` 为空时表示匹配全部流量。

### 6.2 规则语义

- **ALLOW**：匹配 `rules` 的请求被允许，未匹配的可被后续策略拒绝或按默认策略处理。
- **DENY**：匹配 `rules` 的请求被拒绝。
- 每条规则中，`from` 与 `to` 均为「与」关系；规则之间为「或」关系。
- 空 `from` 或空 `to` 表示该维度匹配全部（如无 `from` 即任意来源）。

### 6.3 列表接口

通过 Dynamic Client 列出指定命名空间（或全部）的 AuthorizationPolicy，解析 `spec.action`、`spec.selector.matchLabels`、`spec.rules`，将 `from.source`（ipBlocks、namespaces、principals）与 `to.operation`（hosts、ports、paths、methods）还原为 API 响应结构。

### 6.4 应用与删除

- **应用**：根据请求构建 AuthorizationPolicy 并 Create 到集群；若资源已存在则 Update 覆盖。
- **删除**：按 `namespace` + `name` 删除指定策略。

Istio/Kmesh 将 AuthorizationPolicy 下发到 Envoy，在入口侧执行访问控制决策。

## 7. 集群节点与 Service

### 7.1 `GET /api/cluster/nodes`

使用 `clientset.CoreV1().Nodes().List()` 获取 Node 列表，解析：

- `Status.Conditions` 中 `NodeReady` → 状态
- `Labels` 中 `node-role.kubernetes.io/*` → 角色

使用 `clientset.CoreV1().Services(ns).List()` 获取 Service 列表，返回 `namespace`、`name`，供熔断等场景选择 `spec.host`（如 `svc.ns.svc.cluster.local`）。

## 8. 指标与配置

### 8.1 指标

通过环境变量 `PROMETHEUS_URL` 连接 Prometheus，执行 Instant Query 获取 Kmesh L4 指标（`kmesh_tcp_*` 等），聚合后返回前端。

### 8.2 Accesslog

Accesslog 通过 **K8s Pod Logs API** 直连获取，不依赖 Prometheus。

- **接口**：`GET /api/metrics/accesslog`，查询参数：`pod`（可选，指定 kmesh pod）、`namespace`（默认 `kmesh-system`）、`tail`（默认 200，最大 2000，拉取最近 N 行日志）
- **实现**：使用 LabelSelector `app=kmesh` 列出目标 Pod，再对每个 Pod 调用 `clientset.CoreV1().Pods(ns).GetLogs(podName, &PodLogOptions{Container: "kmesh", TailLines: &tailLines})` 流式读取 kmesh daemon 容器日志，逐行筛选包含 `accesslog:` 前缀的行，提取其后内容作为单条 accesslog
- **前置条件**：需先执行 `kmeshctl monitoring --all enable`，再执行 `kmeshctl monitoring --accesslog enable`，并产生 TCP 流量（如 sleep→tcp-echo）后才有 accesslog 输出
- **响应**：`{ entries: [{ pod, node, content }], podsQueried, message }`

### 8.3 配置

`GET /api/config` 返回 `KIALI_URL`，供拓扑页跳转 Kiali。

## 9. 认证与权限

- 登录成功后签发 JWT，前端在 `Authorization: Bearer <token>` 中携带。
- `AuthMiddleware` 解析 token，将 `user`、`role` 写入 context。
- Casbin 按 `policy.csv` 做 RBAC：`role` 对 `resource`（cluster/waypoint/circuitbreaker/ratelimit/authorization 等）的 `action`（read/write/delete）。
- 写/删操作会调用 `Audit()` 记录审计日志。

| 模块 | 集群资源 | 关键字段 | 依赖 |
|------|----------|----------|------|
| Waypoint | Gateway (gateway.networking.k8s.io) | `gatewayClassName: istio-waypoint`，`istio.io/waypoint-for`，`sidecar.istio.io/proxyImage` | — |
| 熔断 | DestinationRule (networking.istio.io) | `spec.host`，`spec.trafficPolicy.connectionPool`（tcp/http） | 作用于 Waypoint，下发前需检查命名空间已安装 Waypoint |
| 限流 | EnvoyFilter (networking.istio.io) | `configPatches` 注入 `local_ratelimit`，`token_bucket`，`workloadSelector` | 作用于 Waypoint，下发前需检查命名空间已安装 Waypoint |
| 认证策略 | AuthorizationPolicy (security.istio.io) | `spec.action`（ALLOW/DENY），`spec.selector`，`spec.rules`（from/to） | — |

Dashboard 后端不直接操作 Envoy 配置，而是通过标准的 Istio/Gateway API CRD 下发策略，由 Istio/Kmesh 控制面完成到数据面的转换与下发。

## 12. 与 Kmesh 规范对齐及未来计划

参考 [Kmesh 官方文档](https://kmesh.net/zh/docs/welcome) 及项目内 `docs/ctl/`、`docs/proposal/` 规范。

### 12.1 当前实现与规范

| 模块 | 规范来源 | 字段对齐 | 说明 |
|------|----------|----------|------|
| **Waypoint** | `kmeshctl waypoint apply` | ✓ | `gatewayClassName: istio-waypoint`、`istio.io/waypoint-for`、`sidecar.istio.io/proxyImage`、`enrollNamespace`、`overwrite`、`waitReady` 与 kmeshctl 一致 |
| **熔断** | `docs/proposal/circuit_breaker.md` | ✓ | `maxConnections`、`maxPendingRequests`、`maxRequests`、`maxRetries`、`connectTimeoutMs`、`maxRequestsPerConnection` 与 Istio/Envoy 映射一致 |
| **限流** | `docs/proposal/local_ratelimit.md` | ✓ | `token_bucket`（max_tokens、tokens_per_fill、fill_interval）、`workloadSelector` 与 EnvoyFilter 配置一致 |
| **认证策略** | Istio AuthorizationPolicy | ✓ | `action`、`selector`、`rules`（from.source、to.operation）与 Istio 标准一致 |

### 12.2 熔断字段与 Istio/Envoy 映射

| Dashboard 字段 | Istio DestinationRule | Envoy 对应 |
|----------------|-----------------------|------------|
| maxConnections | connectionPool.tcp.maxConnections | max_connection |
| maxPendingRequests | connectionPool.http.http1MaxPendingRequests | max_pending_requests |
| maxRequests | connectionPool.http.http2MaxRequests | max_requests |
| maxRetries | connectionPool.http.maxRetries | max_retries |
| connectTimeoutMs | connectionPool.tcp.connectTimeout | connection_timeout_ms |
| maxRequestsPerConnection | connectionPool.http.maxRequestsPerConnection | max_requests_per_connection |

### 12.3 未来计划

1. **熔断：Outlier Detection（异常检测）**  
   提案 `circuit_breaker.md` 中定义了 Outlier Detection：监控异常实例（如 5xx 错误率），将异常实例从负载均衡池中移除。当前 Dashboard 仅支持 connectionPool 连接池配置，**尚未支持 Outlier Detection**。若 Kmesh 后续支持，需扩展 DestinationRule 的 `outlierDetection` 字段。

2. **限流：HTTP QPS 限流**  
   提案 `local_ratelimit.md` 明确 **None-Goals**：不实现 HTTP 请求级限流或 Quota 限流。当前仅支持连接维度本地限流，与 Kmesh 规范一致。

3. **版本兼容**  
   Kmesh 快速入门要求 Istio 1.22+，Waypoint 需 Istio 1.23~1.25。

4. **命名空间标签**  
   Kmesh 管理需 `istio.io/dataplane-mode=Kmesh` 标签；Waypoint 命名空间需 `istio.io/use-waypoint=<name>`。Dashboard 的 `enrollNamespace` 已实现后者。
