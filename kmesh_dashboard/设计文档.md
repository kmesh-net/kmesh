# Kmesh Dashboard 设计文档

## 1. 背景与目标

### 1.1 背景

在与开发者及终端用户的讨论中，反馈表明 Kmesh 的使用门槛较高。例如 Waypoint 的部署涉及 **Namespace、Service、Workload** 三种粒度；部分流量治理策略需要直接编辑 Envoy 相关配置。经社区会议讨论，决定在 Kmesh 项目中引入 **Kmesh Dashboard**，通过交互式界面降低使用门槛。

### 1.2 目标

- **用户友好**：通过直观的 UI 工作流简化 Kmesh 操作。
- **一键/引导式能力**：Waypoint 安装、熔断、限流等支持向导与即时反馈。
- **可观测与拓扑**：服务拓扑图、指标大盘、健康状态可视化。
- **安全与可维护**：内置认证与 RBAC，并提供完整文档与用户指南。

---

## 2. 功能模块与工作流

### 2.1 Waypoint 安装与管理

**目标**：支持 Namespace / Service / Workload 三种粒度的一键或引导式安装，并给出清晰的可视化反馈。

**工作流**：

1. **入口**：Dashboard 顶部或侧边栏「Waypoint」→「安装 / 管理」。
2. **选择粒度**：
   - **Namespace 级**：选择命名空间 → 可选「为该命名空间打标签（enroll-namespace）」→ 一键应用 → 展示该命名空间下默认 waypoint 状态。
   - **Service 级**：选择命名空间 → 选择 Service（列表来自 K8s API）→ 为该 Service 指定 waypoint 名称、镜像等 → 应用 → 列表/拓扑中展示该 Service 是否已挂载 waypoint。
   - **Workload 级**：选择命名空间 → 选择 Workload（Deployment/StatefulSet 等）或 Pod → 指定 waypoint 名称（如 `reviews-v2-pod-waypoint`）→ 应用 → 状态与列表反馈。
3. **反馈**：
   - 调用后端（封装 `kmeshctl waypoint apply/list/status` 或等价 API）后，在 UI 上显示「安装中 / 就绪 / 失败」及简要原因。
   - 提供「Waypoint 列表」页：按命名空间筛选，表格展示 name、粒度（namespace/service/workload）、状态、创建时间等。

**数据流**：UI → Dashboard 后端 API → Kubernetes API + Kmesh 相关 CR/Label → 可选 kmeshctl 或直接操作 CR。

---

### 2.2 服务拓扑图（类 Kiali）

**目标**：交互式服务拓扑图，展示服务依赖、流量走向与健康状态。

**工作流**：

1. **入口**：「拓扑」或「服务拓扑」菜单。
2. **范围选择**：按命名空间、时间范围、可选按服务/版本过滤。
3. **渲染**：
   - 节点：服务（或 服务:版本），可区分有/无 waypoint、是否在 mesh 内。
   - 边：调用关系，可展示 RPS、错误率、延迟（P50/P99）等。
   - 颜色/图标：健康（绿）、告警（黄）、异常（红）、未知（灰）。
4. **交互**：点击节点 → 侧边栏或抽屉展示该服务详情（Waypoint 状态、最近指标、关联策略）；点击边 → 展示该条调用链的流量与错误率。
5. **数据来源**：Kubernetes 服务/工作负载发现 + 可观测数据（Prometheus/集群内指标），通过 Dashboard 后端聚合后提供给前端。

---

### 2.3 熔断配置

**目标**：简化熔断配置，提供预设模板与实时校验，避免直接编写 Envoy 配置。

**工作流**：

1. **入口**：「流量治理」→「熔断」。
2. **选择作用对象**：按命名空间 + 服务/主机集（Cluster）选择上游。
3. **配置方式**：
   - **预设模板**：如「保守 / 标准 / 激进」等，对应不同的 `maxConnections`、`maxPendingRequests`、`maxRequests`、`maxRetries` 等（与 proposal 中 Envoy/Istio 字段对齐）。
   - **自定义**：表单填写上述阈值 + 可选连接超时、每连接最大请求等；前端做合法性校验（数字范围、必填项）。
4. **实时校验**：提交前或保存前，前端或后端校验参数；保存后展示「已应用」状态，并可关联到拓扑上对应边或节点。
5. **展示**：列表展示已配置的熔断策略（目标、阈值、状态），支持编辑/删除。

**数据流**：UI 表单 → Dashboard 后端 → 转换为 Istio/Envoy 可用的 DestinationRule 或 Kmesh 对应 CR → 写入集群；读取时从集群同步到 UI。

---

### 2.4 限流配置

**目标**：通过可视化策略构建器配置限流规则，并对已生效规则给出即时反馈。

**工作流**：

1. **入口**：「流量治理」→「限流」。
2. **策略构建**：
   - 选择作用对象（命名空间、服务、或更细粒度路由/标签）。
   - 配置限流维度：如 QPS、并发连接数、按 header/路径等。
   - 可视化展示「条件 + 限制值」组合，支持多条规则并列。
3. **即时反馈**：规则应用后，在列表/详情中显示「已下发」；可选在拓扑或指标中观察限流触发情况（如 429 数量）。
4. **数据流**：UI → 后端 → EnvoyFilter 或 Istio EnvoyFilter / 其他 CR → 集群；状态从集群回显。

---

### 2.5 指标大盘

**目标**：集成关键服务网格指标（延迟、错误率、吞吐等），统一展示。

**工作流**：

1. **入口**：「指标」或「可观测」→「服务网格指标」。
2. **内容**：
   - 全局或按命名空间：请求量（RPS）、错误率（4xx/5xx）、延迟（P50/P95/P99）。
   - 可选：按服务、版本、Waypoint 维度下钻。
3. **数据来源**：Prometheus（或集群内已有指标导出），Dashboard 后端做查询与聚合，前端用图表库展示（时序图、柱状图等）。
4. **交互**：时间范围选择、刷新、可选与拓扑/熔断/限流页面联动（如从拓扑点进某服务后跳转到该服务的指标）。

---

### 2.6 认证与 RBAC

**目标**：Dashboard 自身支持登录与基于角色的访问控制，保证安全访问。

**工作流**：

1. **登录**：首次访问重定向到登录页；支持 OIDC / 静态账号密码 / 与 K8s 集群账号集成等（可分期实现）。
2. **角色与权限**：
   - 角色示例：只读（仅查看拓扑、指标、配置）、编辑（可改熔断/限流/Waypoint）、管理员（含用户与权限管理）。
   - 权限与「命名空间」可挂钩：如仅允许某角色访问 `default`、`kmesh-system`。
3. **鉴权**：所有请求经 Dashboard 后端，后端根据当前用户角色与请求资源做校验；前端根据角色隐藏/禁用敏感操作（如删除 Waypoint、编辑熔断）。
4. **审计**：关键操作（安装 Waypoint、修改策略）可记审计日志（后端落库或写集群 Event）。

---

### 2.7 文档与用户指南

**目标**：为上述能力提供完整文档与引导。

**工作流**：

1. **入口**：Dashboard 内「帮助」或「文档」入口；各功能页提供「？」或「引导」链接。
2. **内容**：各模块的「概念说明 + 操作步骤 + 常见问题」；与 Waypoint 三粒度、熔断阈值含义、限流维度等对应。
3. **形式**：站内 Markdown 渲染或链接到外部文档站；新用户首次进入可弹出「快速导览」。

---

## 3. Dashboard 子目录设计（当前实现）

`kmesh_dashboard` 采用前后端分离、文档与配置集中管理的结构，TypeScript + React 前端对接 Go 后端及 Kubernetes API。

```
kmesh_dashboard/
├── README.md                    # 项目说明、本地运行与构建
├── 设计文档.md                  # 本文档
│
├── frontend/                    # 前端（TypeScript + React + Vite）
│   ├── src/
│   │   ├── api/                 # 后端 API 封装
│   │   │   ├── client.ts        # 请求封装
│   │   │   ├── config.ts        # KIALI_URL 等配置
│   │   │   ├── docs.ts          # 站内文档列表与内容
│   │   │   ├── cluster.ts       # 集群节点
│   │   │   ├── waypoint.ts      # Waypoint CRUD
│   │   │   ├── circuitbreaker.ts
│   │   │   ├── ratelimit.ts
│   │   │   ├── metrics.ts
│   │   │   └── services.ts
│   │   ├── components/
│   │   │   └── layout/          # Layout、顶部导航
│   │   ├── pages/
│   │   │   ├── cluster/         # 集群节点
│   │   │   ├── waypoint/        # Waypoint 列表、安装
│   │   │   ├── topology/        # 服务拓扑（跳转 Kiali）
│   │   │   ├── circuitbreaker/  # 熔断列表、表单
│   │   │   ├── ratelimit/       # 限流列表、表单
│   │   │   ├── metrics/         # 指标大盘（ECharts）
│   │   │   └── help/            # 站内文档（Markdown 渲染）
│   │   ├── types/               # cluster、waypoint、metrics 等 TS 类型
│   │   ├── App.tsx
│   │   ├── main.tsx
│   │   └── index.css
│   ├── public/
│   ├── package.json
│   ├── tsconfig.json
│   └── vite.config.ts
│
├── backend/                     # 后端（Go，对接 K8s、Prometheus）
│   ├── cmd/server/             # 入口 main.go
│   ├── internal/
│   │   ├── handler/             # HTTP 路由
│   │   │   ├── handler.go       # 路由注册
│   │   │   ├── cluster.go       # 节点列表
│   │   │   ├── waypoint.go      # Waypoint CRUD
│   │   │   ├── circuitbreaker.go
│   │   │   ├── ratelimit.go
│   │   │   ├── metrics.go       # Prometheus 指标聚合
│   │   │   ├── docs.go          # 站内文档 API（/api/docs）
│   │   │   ├── kiali.go         # Config/KIALI_URL
│   │   │   ├── services.go
│   │   │   ├── accesslog.go
│   │   │   └── cors.go
│   │   └── k8s/                 # K8s client、config
│   ├── go.mod
│   └── Dockerfile
│
├── docs/                        # Dashboard 功能文档（用户指南，站内 Markdown 渲染）
│   ├── waypoint.md
│   ├── topology.md
│   ├── circuitbreaker.md
│   ├── ratelimit.md
│   └── metrics.md
│
└── deploy/                      # 部署相关
    ├── kustomization.yaml
    ├── deployment.yaml
    ├── service.yaml
    └── rbac.yaml
```

**说明**：

- **frontend**：按功能划分 `pages`，与 2.1～2.7 模块对应；`api` 统一走后端；拓扑页跳转集群内 Kiali；帮助页通过 `/api/docs` 拉取 Markdown 并渲染。
- **backend**：负责集群访问、指标查询（Prometheus）、Waypoint/熔断/限流 CR 读写；`handler` 集中 HTTP 逻辑，`k8s` 为集群客户端；访问由 kubeconfig 控制。
- **docs**：存放各功能的用户向说明，由 `/api/docs` 提供，前端 `help` 页站内 Markdown 渲染。
- **deploy**：Kustomize + Deployment/Service/RBAC，便于 K8s 一键部署。

## 4. 技术栈（当前实现）

| 需求           | 实现 |
|----------------|------|
| 语言/框架      | TypeScript + React |
| 构建/开发      | Vite |
| UI 组件        | Ant Design |
| 拓扑图         | 跳转 Kiali（`KIALI_URL` 配置） |
| 图表           | ECharts（echarts-for-react） |
| 状态管理       | React 组件状态 |
| 后端           | Go，REST API |
| K8s 交互       | client-go + Gateway API + dynamic client |
| 访问控制       | kubeconfig（已移除站内 RBAC 登录） |
| 站内文档       | react-markdown + remark-gfm，后端 `/api/docs` 提供 Markdown |

---

## 5. 里程碑建议（不纳入本文档强制范围）

- **M1**：Waypoint 安装与管理（三粒度）+ 列表/状态展示。
- **M2**：服务拓扑图（基础版）+ 指标大盘（核心指标）。
- **M3**：熔断与限流配置界面 + 预设模板与校验。
- **M4**：认证与 RBAC + 用户指南与部署文档。

以上为 Kmesh Dashboard 的初步设计，各组件工作流与目录结构可在实现过程中按反馈迭代调整。
